## 0. 摘要

当前仓库已经形成 v0.1-alpha 的本地 Web 控制台和 Kuavo ROS 包装层。日常入口是 Windows PowerShell 脚本 `scripts/windows/start-kuavo5w-web-control.ps1`，它启动本机 `127.0.0.1:8765` 的 Web 控制台；Web 后端位于 `kuavo_sim_platform/web_control/server.py`，使用 Python 标准库 `ThreadingHTTPServer`，不是 Flask/FastAPI。

Web 控制台已经能列出 YAML 场景、导入 `.py` 脚本、串行执行控制动作、查看 MuJoCo 日志、检查 ROS 接口、运行 smoke test 和采集环境快照。执行链路主要是：浏览器调用 `/api/action` 或 `/api/scripts/import`，后端调用 PowerShell 脚本或 WSL/Docker 命令，再进入固定容器 `kuavo5w_sim`，source ROS/Kuavo 环境后运行 YAML 场景或 Python 脚本。

当前缺少 v0.2 episode 化需要的持久化实验目录、统一 episode runner、metrics/logs/config 副本、操作者身份、任务历史、任务队列、异步状态查询、权限模型和标准化结果 schema。`.gitignore` 已覆盖 `logs/`、`outputs/`、视频、数据集、checkpoint、环境快照和备份目录，但仓库当前没有实际 `outputs/` 或 episode-like 输出目录。

```text

  ↓

  ↓

  ↓

  ↓

  ↓

  ↓

  ↓
```

### 3.4 当前已有安全机制

| 机制 | 当前状态 |
|---|---|
| 默认本机绑定 | `server.py --host` 默认 `127.0.0.1`；Windows 启动脚本固定 `--host 127.0.0.1` |

### 3.5 对 v0.2 episode 化的影响

当前最适合接 episode runner 的位置是 `server.py` 的 `action("scenario")` 和 `action("script_fast"/"script")` 前后，或抽出一层 `episode_runner` 包装现有 `platform()`/`quick_script()`。原因是 Web 路由、串行锁、脚本/场景路径验证、stdout/stderr/exit code 都集中在这里。

当前不适合直接把 episode 逻辑塞进前端或单个 demo 脚本；否则难以统一保存 `config.yaml`、`stdout.log`、`stderr.log`、`metrics.json`、脚本副本、起止时间和操作者信息。

## 4. 脚本执行链路

### 4.1 当前流程

```text
上传
  浏览器读取本地 .py 内容
  POST /api/scripts/import

保存
  server.py validate_script_name()
  MAX_IMPORT_BYTES 检查
  写入 kuavo_sim_platform/imported_scripts/<name>
  返回 sha256/mtime/category

执行
  前端默认调用 /api/action?name=script_fast&script=<name>
  server.py script_container_path() 验证脚本已存在

  container_python() source ROS/Kuavo 环境
  readiness check: /cmd_vel + /mobile_manipulator_mpc_control
  python3 '<container_script_path>'

返回
  subprocess stdout/stderr 合并
  returncode -> code
  ok = returncode == 0
  ts = 后端完成时格式化时间
  前端 <pre> 显示结果，不持久化
```

### 4.3 当前缺口

| 缺口 | 影响 | v0.2 建议 |
|---|---|---|
| 无持久化运行历史 | 前端刷新后结果消失，无法审计 | 每次 run 生成 `outputs/episodes/<timestamp>_<name>_<id>/` |
| stdout/stderr 合并 | 难以自动分析错误 | 分离 `stdout.log`、`stderr.log`，JSON 摘要保留 tail |
| 无 config 副本 | 无法复现实验 | 保存 YAML/脚本/参数副本 |
| 无 metrics.json | 后续分析无统一入口 | 定义 `metrics.json` 和 `result.json` |
| 无操作者身份 | B/C 权限和责任边界不清 | 请求层增加 operator 字段或本地身份配置 |
| Web action 同步阻塞 | 长任务会占用请求，前端只有忙/完成 | 引入 job id + status polling |
| 只有进程内锁 | 重启后无任务状态，无法排队 | 用文件锁/SQLite/job registry |
| 无危险脚本静态检查 | 任意 Python 可导入/执行 | v0.2 首版建议 YAML 优先，Python 上传降级为 A 权限 |
| 无 safe stop 编排层 | 依赖脚本或 SDK 自己收尾 | episode runner finally 中调用 restore/safe stop 策略 |
| 无统一 timeout 策略 | action timeout 分散在代码中 | episode config 定义 timeout，runner 强制上限 |

## 9. 安全机制

| 机制 | 当前状态 | 代码位置 | 风险 | v0.2 建议 |
|---|---|---|---|---|
| 文件名限制 | ASCII `.py`，首字符字母数字 | `server.py SCRIPT_NAME_RE` | 只能防路径穿越/奇怪文件名，不能防危险内容 | 保留，并加内容审查/权限 |
| 文件大小限制 | 512 KiB | `server.py MAX_IMPORT_BYTES` | 可防大文件上传，不防危险脚本 | 保留 |
| 请求大小限制 | 512 KiB + 4096 | `do_POST()` | 合理 | 保留 |
| 脚本路径白名单 | 必须存在于 imported_scripts | `script_container_path()` | 防任意路径，但任意导入脚本仍可执行 | 保留，配合权限 |
| 命令串行执行 | 进程内 lock | `COMMAND_LOCK`/`run_serialized()` | 无持久状态、无队列、重启丢失 | v0.2 用 job registry + file lock/DB |
| 控制锁是否已删除 | 未见旧“控制锁”业务文件；当前为内存命令锁 | `server.py` | 多进程/重启无效 | 设计明确锁语义 |
| timeout | subprocess timeout 分散存在 | `run_command()` 调用处 | 不是 episode 级策略 | 统一到 runner |
| safe stop | SDK `fail_safe()`、restore 脚本、demo_stop | `client.py`、`kuavo5w-restore.ps1` | 任意脚本若不使用 SDK 可能绕过 | runner finally 强制 restore/safe stop |
| 危险脚本阻止 | 无静态阻止 | UNKNOWN | 任意 Python 可调用系统命令/ROS 发布 | B/C 禁止任意 Python 或只允许 YAML |
| 导入路径限制 | 有保存目录和运行路径限制 | `IMPORTED_SCRIPT_DIR` | Python 内部仍可访问系统 | 容器/权限隔离另设 |
| Python import 限制 | 无 | UNKNOWN | 可 import subprocess/os/rospy 发布任意 topic | A-only 任意脚本 |
| B/C 禁止事项文档 | 有当前/归档文档 | `docs/operator-guide.md`、`docs/v0.1_recovery_checklist.md`、`docs/archive/README_remote_collaboration_BC.md` | 分散，非机器可读 | v0.2 形成权限矩阵 |
| 对外网开放 | 默认不开放 | server 默认和启动脚本为 `127.0.0.1` | 若手动 `--host 0.0.0.0` 会开放 | 固定本机或加认证 |
| pause/resume | 暴露 Web 按钮 | `action("pause"/"resume")` | B/C 误操作可能影响实验 | A-only 或二次确认 |
| stop_sim | 暴露 Web 按钮 | `action("sleep"/"stop_sim")` | 会停止仿真 | A-only |

## 13. v0.2 设计约束总结

### 13.1 已具备能力

- 本机 Web 控制台入口和 API dispatch。
- YAML 场景发现、解析、基础校验和运行。
- Python 脚本上传、hash/mtime/size/category 元信息和运行。
- Windows -> WSL -> Docker -> ROS/MuJoCo 的启动和执行链路。
- 固定容器名、固定 image、固定 robot version、固定 launch 默认值。
- 基于 `/cmd_vel` 和 `/mobile_manipulator_mpc_control` 的 readiness 检查。
- MuJoCo 日志 tail、ROS 接口检查、smoke test、环境快照。
- SDK 层 `KuavoSim` 包装 base/arm/lower/head/state，含上下文 fail-safe。
- Web 命令串行锁和上传文件名/大小限制。
- B/C 相关文档和 safe probe 雏形。

### 13.2 缺失能力

- episode runner 和 episode 状态机。
- `outputs/episodes/<run_id>/` 标准目录。
- `config.yaml`、`metrics.json`、`result.json`、`stdout.log`、`stderr.log`、脚本/YAML 副本。
- start/end/duration、operator、task_name、权限等级。
- 任务队列、job id、异步状态查询、最近任务/最近日志/最近结果持久化。
- YAML schema/version、success criteria、全局 timeout、artifact policy。
- Python 脚本危险行为限制或审批流程。
- B/C/A 权限的机器可读策略。
- 失败后的统一 safe stop/finally 编排。

### 13.3 不应轻易改动的部分

- `scripts/windows/start-kuavo5w-platform.ps1` 的 Windows/WSL/Docker/MuJoCo 启动链路；它承载大量环境兼容逻辑。
- 容器名 `kuavo5w_sim`、workspace `/root/kuavo_ws`、deploy mount `/root/kuavo_deploy`，这些已被多处脚本/文档依赖。
- `KuavoSim` 的 fail-safe、base limit、mode switching 逻辑。
- 当前 v0.1 文档归档和备份目录。
- Web 的默认 `127.0.0.1` 绑定。

### 13.4 最适合 v0.2 第一版接入的点

- 以 YAML scenario 为主实验入口，先支持 `base_probe`、`base_forward` 这类低风险模板。
- 在 `server.py action("scenario")` 和 `action("script_fast")` 外层增加 episode wrapper。
- 新增 `outputs/episodes/`，每次运行创建独立目录。
- 把 `collect_env_snapshot.py` 和 `inspect_interfaces.py` 接入 preflight/postflight artifacts。
- 先实现同步 runner + 持久化结果，再扩展异步 job status。
- B/C 先只允许 read-only probe 和白名单 YAML；任意 Python 上传保留给 A。

### 13.5 风险点

- 当前 Web action 是 GET 触发状态改变，不利于审计和权限控制。
- `script_fast` 绕过平台脚本的完整启动流程，只做简短 readiness；适合快，但 episode 化要谨慎。
- 任意 Python 上传能执行系统命令或 ROS 发布，B/C 低风险运行时不应默认开放。
- `COMMAND_LOCK` 只是进程内锁，不能表达队列和历史。
- `stop_sim`/`sleep`、pause/resume 暴露在 Web UI，需权限或确认。
- 当前没有真实运行 `rostopic list` 采样；运行时 topic/service 细节仍需在授权窗口确认。
